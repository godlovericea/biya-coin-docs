## 二、WBS任务分解树

### 层级结构说明
```
Level 0: 项目
  ├─ Level 1: 一级模块（功能模块）
  │   ├─ Level 2: 二级任务（开发/测试阶段）
  │   │   ├─ Level 3: 三级任务（具体工作项）
  │   │   │   └─ Level 4: 四级任务（最小执行单元，0.5-1天）
```

## 五、任务优先级（MoSCoW）

### Must Have（必须有 - P0）
核心功能，MVP必需，延期会影响项目目标

- [ ] 用户登录/注册
- [ ] 核心业务流程
- [ ] 数据安全机制

### Should Have（应该有 - P1）
重要但不紧急，可以在后续版本补充

- [ ] 用户个人信息编辑
- [ ] 操作日志记录
- [ ] 数据导出功能

### Could Have（可以有 - P2）
锦上添花，资源富余时做

- [ ] 个性化主题
- [ ] 快捷键支持
- [ ] 操作引导动画

### Won't Have（不会有）
明确排除，避免范围蔓延

- ❌ 社交分享功能（下期）
- ❌ AI推荐功能（需独立立项）

## 八、任务拆解检查清单

在完成任务拆解后，使用此清单自检：

- [ ] **颗粒度**：最小任务单元是否在0.5-3天之间？
- [ ] **SMART原则**：每个任务是否具体、可衡量、可实现、相关、有时限？
- [ ] **完成标准**：每个任务是否有明确的DoD（Definition of Done）？
- [ ] **依赖关系**：任务间的依赖是否已清晰标注？
- [ ] **关键路径**：是否识别出关键路径并重点关注？
- [ ] **责任人**：每个任务是否有唯一负责人？
- [ ] **工时合理性**：工时评估是否使用了三点估算法？
- [ ] **缓冲时间**：是否预留了15-25%的缓冲时间？
- [ ] **优先级**：是否用MoSCoW标注了优先级？
- [ ] **风险识别**：高风险任务是否有应对预案？
