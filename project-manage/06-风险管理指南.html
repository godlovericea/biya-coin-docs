    风险管理基础

    什么是项目风险？

    定义：可能对项目目标产生负面影响的不确定事件或条件。

    风险三要素：

    风险 = 不确定性事件 × 发生概率 × 影响程度

    风险管理流程

    1. 风险识别   →  找出潜在风险
2. 风险评估   →  分析概率和影响
3. 风险应对   →  制定应对策略
4. 风险监控   →  持续追踪和调整

    为什么要做风险管理？

    不做风险管理 做风险管理
    问题发生后才反应 提前预防和准备
    措手不及，被动救火 从容应对，主动控制
    项目经常延期或失败 项目成功率大幅提升
    团队压力大，士气低 团队有安全感，士气高

    风险识别

    识别方法

    1. 头脑风暴法

    组织团队会议，让所有人提出可能的风险

    会议流程：

    1. 说明会议目的（15分钟）
2. 自由发言，不评判（30分钟）
3. 分类整理（15分钟）
4. 投票排序（10分钟）

    2. 检查清单法

    基于历史项目，制作风险检查清单

    示例清单：

    • 需求是否频繁变更？
    • 团队是否对技术栈不熟悉？
    • 是否依赖外部团队/第三方？
    • 关键人员是否可能离职？
    • 测试环境是否就绪？

    3. 专家访谈法

    咨询有经验的架构师、技术专家

    访谈问题：

    • "您觉得这个项目最大的技术风险是什么？"
    • "以往类似项目遇到过什么问题？"
    • "如果您来做，会在哪些地方投入更多精力？"

    4. SWOT分析法

    Strengths（优势）       Weaknesses（劣势）
- 团队技术能力强        - 对业务领域不熟悉
- 有类似项目经验        - 测试资源不足

Opportunities（机会）   Threats（威胁）
- 可复用公司组件库      - 竞品可能先上线
- 管理层高度重视        - 第三方服务不稳定

    风险评估

    概率-影响矩阵

            影响程度
        低    中    高
概率 高  🟡   🟡   🔴
    中  🟢   🟡   🟡
    低  🟢   🟢   🟡

🟢 低风险：接受并监控
🟡 中风险：制定应对计划
🔴 高风险：立即采取行动

    风险评分计算

    方法1：简单评分法

    风险分数 = 概率分数(1-5) × 影响分数(1-5)

0-5分：  🟢 低风险
6-12分： 🟡 中风险
13-25分：🔴 高风险

    方法2：加权评分法

    考虑多维度影响：

    风险项 概率 进度影响 成本影响 质量影响 综合得分
    (1-5) (1-5)×30% (1-5)×30% (1-5)×40% 
    综合得分 = 概率 × (进度影响×30% + 成本影响×30% + 质量影响×40%)

    风险应对

    应对策略（4T原则）

    1. Transfer（转移）

    将风险转移给第三方

    适用场景：

    • 专业性强，团队不擅长
    • 成本可控，外包更经济

    示例：

    • 购买云服务而非自建服务器（转移运维风险）
    • 使用第三方支付而非自研（转移合规风险）
    • 外包设计工作（转移设计风险）

    2. Tolerate（容忍）

    接受风险，不采取主动措施

    适用场景：

    • 风险概率极低
    • 影响可控
    • 应对成本高于风险本身

    示例：

    • 极低概率的硬件故障
    • 小范围的性能波动
    • 非核心功能的小Bug

    注意：必须准备应急预算和Plan B

    3. Treat（处理）

    采取措施降低概率或影响

    降低概率：

    • 技术预研：提前验证新技术
    • 培训学习：提升团队能力
    • 增加自动化：减少人为错误

    降低影响：

    • 增加缓冲时间：延期影响变小
    • 准备备份方案：快速切换
    • 模块化设计：隔离风险

    示例：

    风险：团队对React Query不熟悉
处理措施：
1. 安排技术分享（降低概率）
2. 结对编程（降低概率）
3. 准备回退到传统fetch方案（降低影响）

    4. Terminate（终止）

    改变计划，避开风险

    适用场景：

    • 风险太大，无法接受
    • 有替代方案

    示例：

    • 不使用不成熟的新框架，选成熟方案
    • 不依赖不稳定的第三方服务，自研简化版
    • 砍掉风险过高的功能需求

    应对计划模板

    ## 风险应对计划

### 风险ID：R001
### 风险描述：第三方支付API不稳定

**当前状态**：🔴 高风险（概率70%，影响高）

**触发条件**：
- 测试环境调用成功率低于95%
- 响应时间超过3秒
- 连续2天出现故障

**主要应对策略**：Treat（处理）

**具体措施**：

1. **预防措施**（降低概率）
   - [ ] 提前2周开始集成测试（而非1周）
   - [ ] 与对方技术团队建立直连沟通渠道
   - [ ] 每日监控接口稳定性指标
   - 责任人：王五
   - 完成期限：11月10日

2. **降低影响措施**
   - [ ] 准备Mock接口，开发可以不依赖真实环境
   - [ ] 增加重试机制和超时处理
   - [ ] 准备降级方案：只上线微信支付，支付宝延后
   - 责任人：王五 + 技术负责人
   - 完成期限：11月12日

3. **应急预案**
   - 如果11月15日前仍不稳定：
     - Plan A：延期上线3天
     - Plan B：先上线微信支付
     - Plan C：联系其他支付服务商
   - 决策人：项目经理 + 技术负责人

**预算**：
- 额外人力成本：2人天（准备降级方案）
- 时间成本：可能延期0-3天

**跟进频率**：每日

**最后更新**：2025-11-07

    风险监控

    监控机制

    1. 每日站会关注风险

    每日站会提问：
1. 昨天完成了什么？
2. 今天计划做什么？
3. 遇到什么阻碍？ ← 关注点：这是风险信号

    2. 周报追踪风险变化

    风险状态变化追踪：

上周  本周  变化  说明
🔴 → 🟡  ↓   第三方API问题已部分解决
🟡 → 🟡  →   需求变更风险持续存在
🟢 → 🟡  ↑   新发现测试资源不足

    3. 风险仪表盘

    项目风险总览（第5周）

🔴 高风险：1个 （↓ 较上周减少1个）
🟡 中风险：3个 （→ 与上周持平）
🟢 低风险：5个 （↑ 较上周增加2个）

总体趋势：🟢 好转

    风险触发器（Trigger）

    设置明确的触发条件，自动预警

    风险项 触发条件 触发时行动
    进度延期 实际进度落后计划>5% 启动加班或增加人力
    Bug爆发 P0/P1 Bug超过10个 暂停新功能，全员修Bug
    人员流失 核心成员提离职 启动知识转移，寻找替补
    性能问题 接口响应时间>1秒 暂停功能开发，优先优化

    风险审查会议

    频率：每2周一次

    参与人：项目经理、技术负责人、核心开发

    议程：

    1. 回顾风险登记表（15分钟）
    2. 评估已识别风险的状态变化（20分钟）
    3. 识别新风险（15分钟）
    4. 更新应对计划（10分钟）

    常见风险清单

    技术风险

    风险 典型场景 预防措施 应急预案
    新技术不成熟 使用刚发布的框架 提前POC验证 准备回退到成熟技术
    性能不达标 高并发场景 提前压测 限流、降级、扩容
    第三方服务不稳定 依赖外部API 沙盒环境测试 Mock、重试、降级
    技术选型错误 架构设计失误 架构评审 重构或替换
    安全漏洞 SQL注入、XSS 代码审查、安全测试 紧急修复+回滚
    数据丢失 数据库故障 定期备份 恢复备份数据

    管理风险

    风险 典型场景 预防措施 应急预案
    范围蔓延 需求不断新增 需求冻结机制 砍需求或延期
    进度延期 低估工作量 三点估算+缓冲 加班或减功能
    沟通不畅 跨团队协作 每日同步会议 升级到管理层协调
    资源冲突 人员被其他项目占用 提前资源预定 协商优先级
    计划变更 公司战略调整 保持计划灵活性 快速响应调整

    人员风险

    风险 典型场景 预防措施 应急预案
    核心人员离职 技术大牛突然辞职 知识文档化、交叉培养 内部调配或外部招聘
    技能不匹配 团队不会新技术 培训学习 外部技术顾问
    人员请假/生病 关键时期请假 备份人员计划 任务重新分配
    团队冲突 成员矛盾 团队建设 一对一沟通或调岗
    士气低落 长期加班 合理排期、及时激励 调整节奏、团建

    外部风险

    风险 典型场景 预防措施 应急预案
    客户需求变更 产品方向调整 敏捷迭代 快速响应
    竞品先上线 市场竞争 加快迭代速度 差异化策略
    政策法规变化 合规要求 关注政策动态 紧急合规改造
    供应商问题 云服务故障 多云备份 切换到备用服务商

    实战案例

    案例1：第三方API风险应对

    背景：

    • 项目：电商支付系统对接
    • 风险：支付宝API测试环境不稳定

    风险识别：

    • 时间：项目启动第2周
    • 方式：技术预研时发现

    风险评估：

    • 概率：70%（测试环境经常超时）
    • 影响：高（支付是核心功能，无法上线）
    • 风险等级：🔴 高风险

    应对措施：

    策略1：Treat（处理 - 降低概率）
- 与支付宝技术支持建立直连（微信群）
- 每日监控接口稳定性
- 提前3周开始集成测试（原计划2周）

策略2：Treat（处理 - 降低影响）
- 准备Mock接口，不阻塞前端开发
- 增加重试机制（最多3次）
- 增加超时处理（5秒超时降级）

策略3：Terminate（终止 - 备用方案）
- 先上线微信支付（稳定），支付宝延后
- 评估使用其他支付服务商（如聚合支付）

    实际发生：

    • 第4周：支付宝接口仍不稳定（触发条件）
    • 决策：先上线微信支付
    • 结果：项目按时上线，2周后补上支付宝

    经验教训：
    ✅ 提前识别风险，准备了Plan B，避免了延期
    ✅ Mock接口让前端不受阻塞
    ⚠️ 下次应该更早接入，留更多时间

    案例2：核心开发请假的应对

    背景：

    • 项目：关键功能开发期
    • 风险：负责支付模块的后端工程师突然病假1周

    风险识别：

    • 时间：项目第5周（距离上线还有2周）
    • 方式：员工当天早上请假通知

    风险评估：

    • 概率：已发生（100%）
    • 影响：高（支付模块在关键路径上）
    • 风险等级：🔴 紧急事件

    应对措施（实时反应）：

    Day 1（当天）：

    08:00 - 收到请假通知
09:00 - 紧急评估影响：支付模块开发进度60%，还需3天
10:00 - 启动应急预案：
        1. 检查代码和文档
        2. 寻找备选人员
11:00 - 决策：前端组长（曾做过后端）临时接手
14:00 - 知识转移会议（电话）：病假同事讲解思路
17:00 - 前端组长熟悉代码，列出不清楚的点

    Day 2-3：

    - 前端组长完成剩余开发（比原计划慢1天）
- 技术负责人协助Code Review
- 准备测试环境

    结果：

    • 延期1天完成（原计划3天，实际4天）
    • 质量未受影响（Code Review严格）
    • 项目总体进度影响可控

    预防措施（事后改进）：

    1. 建立关键任务备份人员机制
    2. 代码和文档及时更新，降低交接成本
    3. 避免单点依赖，关键模块至少2人熟悉

    案例3：需求频繁变更的应对

    背景：

    • 项目：移动端App开发
    • 风险：产品经理每周都提新需求或改需求

    风险识别：

    • 时间：项目第2周开始频繁发生
    • 方式：每次评审会后都有新想法

    风险评估：

    • 概率：80%（几乎每周都有）
    • 影响：中（增加工作量，影响士气）
    • 风险等级：🟡 中风险（但持续发生会变高风险）

    应对措施（过程改进）：

    第1步：数据化问题

    统计3周数据：
- 新增需求：8个（增加15人天工作量）
- 修改需求：12处（返工10人天）
- 总计影响：25人天（占总工时的20%）

用数据向管理层汇报问题严重性

    第2步：建立变更流程

    需求变更管理流程：

1. 需求提出（产品）
   ↓
2. 影响评估（技术+PM）
   - 工时影响
   - 进度影响
   - 是否影响上线
   ↓
3. 优先级评审（产品+技术+管理层）
   - P0：必须本期做
   - P1：下期做
   - P2：未来考虑
   ↓
4. 更新排期并知会全员

    第3步：需求冻结机制

    里程碑冻结规则：
- 距离上线4周：需求冻结，只接受P0紧急需求
- 距离上线2周：完全冻结，只修Bug
- 距离上线1周：只修P0/P1 Bug

    结果：

    • 后续4周，新需求数量下降70%
    • 团队士气明显改善
    • 项目按时上线

    关键成功因素：
    ✅ 用数据说话，而非主观抱怨
    ✅ 获得管理层支持
    ✅ 建立明确的规则并严格执行

    风险管理工具推荐

    风险登记表工具

    • Excel/Google Sheets：轻量级，适合小项目
    • Jira：可以创建"风险"issue类型
    • Notion：可视化好，适合团队协作
    • 专业工具：RiskyProject（大型项目）

    监控与提醒

    • 项目管理工具：设置风险触发提醒
    • 监控系统：实时监控技术指标（如API响应时间）
    • 自动化脚本：定期检查并发送风险报告

    风险管理最佳实践

    DO（推荐做法）

    ✅ 尽早识别：项目启动时就做风险分析
    ✅ 全员参与：鼓励团队成员提出风险
    ✅ 定期评审：每2周更新风险状态
    ✅ 数据驱动：用数据支持风险判断
    ✅ 准备Plan B：高风险项必须有备选方案
    ✅ 快速响应：风险触发后24小时内启动应对
    ✅ 经验积累：项目结束后总结风险管理经验

    DON'T（避免做法）

    ❌ 鸵鸟心态：假装风险不存在
    ❌ 报喜不报忧：向上汇报时隐瞒风险
    ❌ 过度悲观：把所有事都当风险，导致决策瘫痪
    ❌ 不做记录：口头提过就算，没有文档追踪
    ❌ 只识别不应对：知道有风险但不做准备
    ❌ 风险发生才重视：平时不关注，出事了才救火

    总结

    风险管理核心要点

    1. 识别要早：项目启动时就做
2. 评估要准：量化概率和影响
3. 应对要全：预防+降低+备选
4. 监控要勤：定期检查状态
5. 文档要全：记录在风险登记表

    风险管理成熟度模型

    Level 1 - 初始级：被动救火，风险发生才反应
Level 2 - 可重复级：有风险意识，会做风险登记
Level 3 - 已定义级：有完整流程，定期评审
Level 4 - 已管理级：量化管理，数据驱动决策
Level 5 - 优化级：持续改进，预测性管理

    目标：在3-6个月内从Level 1提升到Level 3

    第一次做风险管理的行动清单

    • 召开风险识别头脑风暴会议
    • 创建项目风险登记表（至少识别5个风险）
    • 对前3大风险制定应对计划
    • 设置风险触发条件和监控机制
    • 在每周周报中追踪风险状态
    • 项目结束后做风险管理复盘

    记住：风险管理不是为了消除所有风险（不可能），而是让风险变得可控、可预测、可应对。
