## ⚡ 工时评估速查

### 三点估算公式
```
期望工时 = (乐观值 + 4×最可能值 + 悲观值) / 6
对外承诺 = 期望工时 × 1.5
```

### 故事点参考
```
1点 = 0.5天  |  修改文案、调整样式
2点 = 1天    |  简单表单、基础组件
3点 = 2-3天  |  CRUD功能
5点 = 4-5天  |  用户认证、权限系统
8点 = 1周    |  支付集成、复杂业务
13点 = 2周   |  需要拆分！
```

### 缓冲时间分配
```
核心开发：60%
测试联调：15%
需求变更：10%
技术风险：10%
会议沟通：5%
```

## 🔴 风险评估矩阵

```
风险分数 = 概率(1-5) × 影响(1-5)

0-5分：  🟢 接受并监控
6-12分： 🟡 制定应对计划
13-25分：🔴 立即采取行动
```

### 应对策略4T
```
Transfer   → 外包、购买保险
Tolerate   → 接受风险，准备预算
Treat      → 降低概率或影响
Terminate  → 改变方案，避开风险
```

## 📅 会议速查

### 每日站会（15分钟）
```
1. 昨天完成了什么？
2. 今天计划做什么？
3. 遇到什么阻碍？
```

### 周报关键内容
```
✅ 本周完成（用数据说话）
🚧 进行中任务（进度百分比）
📅 下周计划（具体任务）
⚠️ 风险与问题（用红黄绿标识）
📊 数据指标（进度、Bug、覆盖率）
```

## 📈 进度跟踪公式

### 进度偏差
```
进度偏差 = 实际进度 - 计划进度
进度偏差率 = 进度偏差 / 计划进度 × 100%

-5%以内：🟢 正常
-5%到-10%：🟡 轻微延期
-10%以上：🔴 严重延期
```

### 预计完成时间
```
剩余工时 = 总工时 × (1 - 实际进度)
预计完成日期 = 当前日期 + 剩余工时/团队人数
```


## 🚨 风险触发条件

```
进度延期     → 实际进度落后>5%
Bug爆发      → P0/P1超过10个
人员流失     → 核心成员提离职
性能问题     → 接口响应>1秒
第三方故障   → 成功率<95%
```

## 🚩 常见错误与避坑

### ❌ 过于乐观的估算
```
错误："这个简单，1天搞定"
正确：使用三点估算，考虑悲观情况
```

### ❌ 计划过于细致
```
错误：2个月项目拆成500个任务
正确：近期详细，远期粗略；颗粒度1-3天
```

### ❌ 忽视沟通成本
```
错误：3人10天的活=1人30天
正确：预留15-25%的沟通协调时间
```

### ❌ 不敢说"不"
```
错误：产品提需求全部答应
正确：走变更流程，评估影响，用数据说话
```

### ❌ 微观管理
```
错误：频繁询问进度，干涉技术细节
正确：信任团队，聚焦结果，关键点深入
```

## 🔢 常用计算公式

### 工期计算
```
理论工期 = 总工时 / 团队人数
实际工期 = 理论工期 × 沟通系数(1.2-1.5)
对外承诺 = 实际工期 × 1.3
```

### 团队效率
```
布鲁克斯定律：向延期项目增加人手会让项目更延期

2人团队沟通路径：1条
3人团队沟通路径：3条
5人团队沟通路径：10条
N人团队沟通路径：N×(N-1)/2

建议：团队不超过7人（2 pizza team）
```

### 偏差系数
```
偏差系数 = Σ(实际工时/预估工时) / 项目数
下次预估 = 初步预估 × 偏差系数
```

## ⚙️ 应急决策树

```
项目延期怎么办？

1. 评估延期原因
   ├─ 工时低估 → 增加缓冲时间
   ├─ 需求变更 → 砍需求或延期
   ├─ 技术难题 → 寻求技术支持
   └─ 人员不足 → 增加人力或延期

2. 可选方案
   ├─ 方案A：加班赶进度
   ├─ 方案B：减少功能范围
   ├─ 方案C：增加人力
   └─ 方案D：延期上线

3. 决策标准
   ├─ 质量不能妥协
   ├─ 核心功能必须有
   └─ 团队健康要保证
```
